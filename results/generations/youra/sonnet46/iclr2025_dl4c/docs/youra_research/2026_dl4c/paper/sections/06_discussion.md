# 7. Discussion

## 7.1 What This Result Establishes

The h-e1 evaluation yields a PARTIAL gate result: infrastructure validated, model behavioral prerequisite unmet. We interpret this outcome as a **prerequisite discovery**, not an experimental failure. The distinction matters: a failure would mean the hypothesis is falsified or the methodology is flawed. A prerequisite discovery means we have identified, with precision, the single missing artifact that blocks evaluation of the hypothesis. Both the infrastructure and the analytical mechanism are sound; the missing piece is an SFT checkpoint.

**Finding 1: The prescreening pipeline is a reusable contribution.** The six-module implementation (prescreening.py, evaluate.py, reward_fn.py, data_loader.py, execution_sandbox.py, visualization.py) constitutes a production-ready prescreening harness for GRPO-for-code research. The pipeline generalizes beyond our specific hypothesis: any researcher wishing to filter problems to a tractability window before applying GRPO can use this infrastructure with a different model, reward function, or target window. The 15/15 task completion and 67/67 test pass rate confirm production quality.

**Finding 2: SFT initialization is a hard prerequisite for GRPO on APPS.** Our observation of 0% pass rate from Qwen2.5-Coder-7B-Instruct on APPS introductory problems under strict execution is consistent with Afterburner [Du et al., 2025] and extends its finding: even an instruction-tuned model with strong HumanEval performance requires an APPS-specific SFT pass before GRPO can proceed. This finding generalizes: **researchers applying GRPO to competitive programming datasets should not assume instruction-tuned models are ready for execution-based reward signals without format alignment**. The SFT prerequisite is a necessary condition, not a suggested best practice.

**Finding 3: The Binomial variance mechanism is analytically sound.** Although we cannot yet confirm the variance advantage of R_ratio empirically (Stage 2 is blocked), the Binomial variance derivation in Section 3.2 and the simulated advantage distributions in Figure 3 establish that the mechanism is coherent. The ratio E[Var(r_ratio)] / E[Var(r_binary)] ≥ 5× for q ∈ [0.3, 0.55], T = 5 is a mathematical consequence of the reward definitions, not an empirical claim. Once qualifying prescreened groups exist, the variance advantage will be observable.

## 7.2 Limitations

**L1 — SFT checkpoint gap (critical).** The single missing artifact is an SFT checkpoint trained on APPS introductory solutions. All five sub-hypotheses (h-e1 quantitative metrics, h-m1 through h-m4) are blocked until this checkpoint is available. *Why acceptable:* The gap is identified, bounded, and actionable — it does not require new methodology, only an additional training run. *Future work:* Train SFT checkpoint on APPS introductory solutions using standard supervised fine-tuning; re-run prescreening stage.

**L2 — Base model format incompatibility (probable cause of 0% pass rate).** The Qwen2.5-Coder-7B-Instruct format produces chat-style outputs incompatible with our raw-execution sandbox. This is the most likely proximate cause of 0% pass rate, alongside genuine difficulty. *Why acceptable:* This is precisely what SFT initialization corrects; the finding motivates rather than undermines our experimental design. *Future work:* A parsing layer that strips markdown fences could serve as a lightweight diagnostic, separating format failure from genuine solve rate.

**L3 — Tractability window assumption unverified.** We assume SFT initialization will bring S_term into the [0.3, 0.55] window for a substantial fraction of APPS introductory problems. If the SFT model achieves >80% pass rate on easy problems, most problems will fall above the window; if it achieves <10%, most will fall below. *Why acceptable:* The window parameters are adjustable — if the SFT checkpoint lands outside [0.3, 0.55], we can widen the window or adjust difficulty tier. *Future work:* Empirically calibrate window bounds after SFT checkpoint is available.

**L4 — Limited problem sample (300/1,923).** We processed 300 of 1,923 available introductory problems. If the qualifying problem rate is low (e.g., 5-10% after SFT), the sample may be insufficient. *Why acceptable:* The current result (0% qualifying) is unambiguous at any sample size. After SFT, we will process the full 1,923 problems. *Future work:* Full-corpus prescreening after SFT.

**L7 — Single model family (intentional design choice).** All experiments use Qwen2.5-Coder-7B. Generalization to other model families (CodeLlama, DeepSeek-Coder, StarCoder2) is not tested. *Why acceptable:* The Binomial variance mechanism is model-agnostic — it depends only on the reward function definition and the pass rate distribution, not on model architecture. The specific quantitative predictions (5–20× variance ratio) will vary by model and problem set, but the direction of the effect is a mathematical consequence. *Future work:* Replicate prescreening-gated GRPO with one additional model family to assess generalization.

## 7.3 Broader Impact

The primary contribution of this work is methodological: a prescreening protocol and reward comparison framework for RL-for-code research. The prescreening pipeline generalizes to any setting where GRPO is applied to execution-based rewards on problems with partial credit structure. The comparison methodology — pre-registering gate metrics and thresholds before running experiments — is a practice we encourage for transparency in RL-for-code research, where reward hacking and selective reporting are active concerns.

We do not foresee significant negative societal impacts from this methodology contribution. The prescreening approach could in principle be used to select training problems that make a model more specialized and less general, but this concern applies to any data curation strategy and is not specific to our approach. The reward comparison framework we propose (R_ratio vs R_binary with variance-based gate metrics) may reduce wasted compute by filtering out problem sets where no useful reward signal exists, which is a positive efficiency outcome.

The SFT prerequisite finding, if it generalizes, has a practical implication for practitioners: teams applying GRPO to coding tasks should invest in format-aligned SFT initialization before attempting GRPO, rather than relying on instruction-tuned base models. This is a constructive, concrete recommendation that emerges directly from the h-e1 result.
