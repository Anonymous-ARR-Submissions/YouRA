# Real GitHub Data Collection for H-E1

## Overview

This experiment validates the hypothesis that human code modifications exhibit aspect-dominant structure across quality dimensions. **This requires REAL GitHub commit data with actual quality metrics** - synthetic/mock data is not acceptable as it would make the experiment tautological.

## Why Real Data Matters

Using synthetic data with hardcoded aspect-to-metric mappings would:
- ❌ Guarantee the hypothesis passes (circular logic)
- ❌ Provide no evidence for real-world separability
- ❌ Invalidate the entire pipeline's empirical foundation

The hypothesis can ONLY be validated with:
- ✅ Real commits from actual GitHub repositories
- ✅ Real quality metrics from SonarQube, CodeQL, pytest, etc.
- ✅ Natural variability in developer behavior

## Data Collection Pipeline

### Phase 1A: Commit Mining (~5 days)

**Goal:** Collect 10,000 minimal-diff commits from GitHub

**Requirements:**
- GitHub API token (increases rate limit from 60 to 5,000 requests/hour)
- PyGitHub, PyDriller, tree-sitter packages
- 500-1000 target repositories (≥1000 stars, active development)

**Filters:**
- AST distance: 1-20 nodes changed (minimal-diff constraint)
- File types: `.py`, `.js`, `.java` source files only
- Time range: 2023-2026 (contemporary practices)
- Aspect labels: security, refactor, performance, bugfix

**Command:**
```bash
export GITHUB_TOKEN=<your_personal_access_token>
python scripts/phase1a_data_collection.py --n-commits 10000
```

**Output:** `data/commits_10k.jsonl` (commit metadata with aspect labels)

---

### Phase 1A: Metric Computation (~22 hours)

**Goal:** Compute 4 quality metrics for each commit

**Requirements:**
- Docker (for SonarQube)
- CodeQL CLI from GitHub releases
- pytest, jest, pytest-benchmark
- 16-core CPU for parallel processing

**Metrics:**

1. **Δcorrectness:** Test pass rate change
   - Run pytest/jest on pre/post file versions
   - Compare test pass rates
   - Handle missing test suites gracefully

2. **Δquality:** SonarQube maintainability rating change
   - Start SonarQube: `docker run -d -p 9000:9000 sonarqube:latest`
   - Run `sonar-scanner` on pre/post versions
   - Extract maintainability rating delta

3. **Δsecurity:** CodeQL alert count change
   - Download CodeQL CLI
   - Analyze pre/post versions
   - Count security alerts (XSS, injection, etc.)

4. **Δefficiency:** pytest-benchmark runtime change
   - Run pytest-benchmark on affected functions
   - Compute % runtime improvement

**Command:**
```bash
# Start SonarQube (required for Δquality)
docker run -d --name sonarqube -e SONAR_ES_BOOTSTRAP_CHECKS_DISABLE=true \
    -p 9000:9000 sonarqube:latest

# Install CodeQL (required for Δsecurity)
wget https://github.com/github/codeql-cli-binaries/releases/latest/download/codeql-linux64.zip
unzip codeql-linux64.zip
export PATH=$PATH:$(pwd)/codeql

# Compute metrics (parallel processing across 16 cores)
python scripts/phase1a_metric_computation.py \
    --commits data/commits_10k.jsonl \
    --output data/outcome_matrix.npy \
    --n-workers 16
```

**Output:** `data/outcome_matrix.npy` (10K×4 NumPy array)

## Expected Data Format

### commits_10k.jsonl
JSONL file with one commit per line:
```json
{
  "commit_id": "abc123...",
  "repo_name": "org/repo-name",
  "message": "Fix security vulnerability in auth module",
  "aspect_label": "security",
  "metrics": {
    "correctness": 0.05,
    "quality": 1.2,
    "security": 3.8,
    "efficiency": -0.1
  }
}
```

### outcome_matrix.npy
NumPy array with shape `(10000, 4)` containing:
- Column 0: Δcorrectness (test pass rate change)
- Column 1: Δquality (SonarQube rating change)
- Column 2: Δsecurity (CodeQL alert count change, negative = more secure)
- Column 3: Δefficiency (runtime % change, positive = faster)
