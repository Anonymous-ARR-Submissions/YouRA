# Combining LLM Code Generation with Formal Specifications and Reactive Program Synthesis

## Section Summaries

### Abstract
In the past few years, Large Language Models (LLMs) have exploded in usefulness and popularity for code generation tasks. However, LLMs still struggle with accuracy and are unsuitable for high-risk applications without additional oversight and verification. In particular, they perform poorly at generating code for highly complex systems, especially with unusual or out-of-sample logic. For such systems, verifying the code generated by the LLM may take longer than writing it by hand. We introduce a solution that divides the code generation into two parts; one to be handled by an LLM and one to be handled by formal methods-based program synthesis. We develop a benchmark to test our solution and show that our method allows the pipeline to solve problems previously intractable for LLM code generation.

### Introduction & Motivation
LLMs have demonstrated strong performance in code generation benchmarks but remain unsuitable for mission-critical systems due to a lack of formal correctness guarantees. Large, complex codebases require manual verification, negating productivity gains. This work addresses the gap by combining LLMs’ natural language capabilities with formal program synthesis to generate code guaranteed to satisfy temporal logic specifications. Specifically, it proposes using LLMs to translate natural language into Temporal Stream Logic (TSL) specifications and then synthesizing correct-by-construction reactive code from these specs. The challenge addressed is balancing LLM flexibility with formal synthesis’s strictness by leveraging TSL’s ability to include “holes” for LLM-generated function and predicate terms, maintaining structural correctness while filling in detailed logic via LLMs.

### Methodology
The core method is a two-stage pipeline that integrates LLM code generation with formal reactive program synthesis via Temporal Stream Logic (TSL):

- **Step 1: NL to TSL Specification Generation**
  - Inputs: 
    - (a) High-level natural language problem summary.
    - (b) Detailed natural language descriptions of assumptions and guarantees.
    - (c) Function and predicate terms defining the separation of data and control.
  - An LLM (GPT-4) is prompted with few-shot in-context learning (ICL) techniques using a structured prompt template with NL-to-TSL examples and explanations of TSL syntax.
  - Output: a TSL formal specification capturing temporal logical requirements of the reactive system.

- **Step 2: Reactive Synthesis**
  - The TSL specification is fed into a formal synthesis engine that generates correct-by-construction Mealy machine implementations.
  - TSL formulas distinguish between predicate terms (\(\tau_P \in T_P\)) used for environmental observations and function terms (\(\tau_F \in T_F\)) used to construct outputs.
  - The realizability problem for TSL is formally stated as:
    \[
    \exists \sigma \in I^+ \to O. \forall \iota \in I^\omega . \forall \langle \cdot \rangle : F \to F, \quad \sigma \frown \iota, \iota \models_{\langle \cdot \rangle} \varphi
    \]
    where \(\sigma\) is a strategy satisfying the specification \(\varphi\).
  - Synthesized code covers the temporal control logic and is guaranteed to meet the specification.

- **Step 3: LLM Code Generation for Data Handling and Integration**
  - The function and predicate "holes" in synthesized code (handling data transformations and low-level logic) are filled using LLM-generated code.
  - This exploits the pure functional separation in TSL to produce wrapper or integration code, reducing the verification burden.
  - Additionally, inspired by Magicoder's prompt seeding, the synthesized code is used to seed LLM prompts to generate entire solutions grounded in correct-by-construction synthesis.

- **Model Architecture and Hyperparameters**
  - Utilizes GPT-4 as the LLM for natural language to TSL translation and function/predicate code generation.
  - Few-shot ICL prompting is employed, with a carefully curated prompt template containing structured NL and TSL examples.
  - Details such as learning rate, batch size, or epochs are not applicable as no conventional training is performed.

- **Input/Output and Data Preprocessing**
  - Inputs: natural language descriptions specifying system behavior, including explicit assumptions and guarantees.
  - Outputs: synthesized reactive system code in software form, with verified temporal logic components and LLM-generated data-handling code.
  - Data preprocessing consists mainly of prompt engineering to map NL to structured formal logic.

- **Novel Components**
  - The integration of TSL’s predicate/function separation with LLM code generation allows a hybrid where verified reactive control logic coexists with flexible data manipulations by an LLM.
  - Employs prompt seeding with synthesized code to boost LLM performance for complex system generation, beyond mere post-processing.
  - Use of TSL as the specification logic rather than standard LTL facilitates modular and practical synthesis for complex reactive systems.

    2. **Complex Arbiter Problems**: scalability tests involving arbiters with 10, 20, and 30 conditions modeling resource-management reactive systems requiring properties like mutual exclusion and eventual grants.

- **Ablation Findings**

  - Prompt seeding boosts complex task success rate substantially compared to naive LLM-only generation.
  - Failures in generating synthesizable TSL specifications indicate limits of few-shot prompting in unfamiliar formal languages.

### Discussion & Conclusion
Combining LLMs with formal reactive program synthesis via TSL enables significant reductions in unverified code lines, improving trustworthiness for safety-critical and complex reactive systems. The flexible function and predicate term abstraction allows LLMs to fill in data logic while temporal control logic remains formally verified. Prompt seeding with synthesized code further enhances LLM efficacy on complex tasks. Limitations include difficulties in generating synthesizable TSL specifications purely via few-shot prompting without fine-tuning. Future work includes integrated fine-tuning pipelines, broader task coverage, and improving the NL-to-TSL translation robustness.
