## Name

failure_aware_code_generation

## Title

Learning from Execution Failures: Contrastive Guidance for Semantic Error Prevention in Code Generation

## Short Hypothesis

Large language models can generate more semantically correct code if they are explicitly shown common execution failure patterns as negative examples during generation, rather than relying solely on post-hoc correction. We hypothesize that augmenting prompts with retrieved failure patterns and their corrections from a curated failure library will reduce semantic errors (logic bugs, incorrect API usage, boundary errors) more effectively than standard few-shot prompting or self-debugging alone.

## Related Work

This proposal distinguishes from existing work in three key ways: (1) Unlike self-debugging approaches (RLEF, CYCLE, PairCoder) that focus on iterative correction after generation, we prevent errors during initial generation through contrastive learning from failures. (2) Unlike constrained decoding methods (TreeCoder, SCodeGen) that enforce syntactic or security constraints, we address semantic correctness through learned failure patterns. (3) Unlike standard RAG for code that retrieves similar successful solutions, we retrieve failure patterns as negative demonstrations. Recent work on contrastive decoding (FCD) focuses on improving text quality through logit manipulation, while we use explicit failure examples as learning signals. The closest related work is property-based testing for validation, but we use execution failures proactively during generation rather than reactively for validation.

## Abstract

Code generation with large language models (LLMs) has made remarkable progress, yet models continue to produce semantically incorrect code with logical errors, incorrect API usage, and boundary condition mistakes. Current approaches address this through post-generation self-debugging or hard syntactic constraints during decoding, but fail to prevent semantic errors during initial generation. We propose Failure-Aware Code Generation (FACG), a novel framework that learns from execution failure patterns to guide LLMs toward semantically correct code generation. FACG constructs a lightweight failure pattern library by executing generated code on test cases, extracting common failure signatures (error types, stack traces, failed assertions), and clustering them into interpretable failure modes. During generation, FACG retrieves relevant failure patterns based on the current task and augments the prompt with contrastive examples showing both the failure and its correction. This provides the model with explicit negative demonstrations of what to avoid. We evaluate FACG on HumanEval, MBPP, and APPS benchmarks using CodeLlama and StarCoder models. Our experiments show that FACG reduces semantic errors by 23-31% compared to standard few-shot prompting and achieves 15-20% higher pass@1 rates than self-debugging baselines, while requiring only a single generation pass. Ablation studies demonstrate that contrastive failure examples are more effective than positive examples alone, and that failure pattern retrieval outperforms random negative sampling. This work establishes a new paradigm for code generation that learns from mistakes proactively rather than reactively, offering a practical and computationally efficient approach to improving semantic correctness.

## Experiments

We will conduct the following experiments: (1) Failure Library Construction: Execute 5,000 diverse code samples from HumanEval, MBPP, and APPS on their test suites, collecting execution traces, error messages, and failed assertions. Cluster failures using embedding similarity (CodeBERT) into 50-100 failure pattern categories (e.g., 'off-by-one in loops', 'incorrect list indexing', 'wrong API parameter order'). For each pattern, store: failure signature, example failing code, corrected code, and natural language description. (2) Main Evaluation: Test FACG on HumanEval, MBPP, and APPS using CodeLlama-7B/13B and StarCoder-15B. For each problem, retrieve top-3 most similar failure patterns using task description embeddings. Compare against: (a) Standard few-shot (3 positive examples), (b) Self-debugging (generate + test + refine), (c) Positive-only RAG (retrieve similar successful solutions). Metrics: pass@1, pass@10, semantic error rate (manual analysis of 200 samples), generation time. (3) Ablation Studies: (a) Vary number of retrieved failure patterns (0,1,3,5), (b) Compare failure-only vs. failure+correction vs. correction-only prompts, (c) Test with random negative examples vs. retrieved relevant failures, (d) Evaluate on different failure categories separately to identify which patterns benefit most. (4) Generalization Test: Train failure library on Python (HumanEval/MBPP), test on Java and C++ problems to assess cross-language transfer. (5) Human Evaluation: Have 3 experienced developers rate 100 generated code samples on semantic correctness, code quality, and whether they exhibit common failure patterns.

## Risk Factors And Limitations

Key risks include: (1) Failure library quality depends on initial code generation diversity - we mitigate this by sampling from multiple models and temperatures. (2) Retrieval accuracy is critical - poor matches may confuse the model rather than help. We will use careful embedding-based retrieval with similarity thresholds. (3) Prompt length limitations - showing multiple failure examples may exceed context windows for smaller models. We will experiment with compressed failure descriptions. (4) The approach may not generalize to entirely novel failure modes not seen during library construction - we will evaluate on held-out problem types. (5) Computational cost of building the failure library requires executing thousands of programs, but this is a one-time offline cost. (6) Manual clustering and annotation of failure patterns requires human effort, though we will explore automated clustering. (7) The approach focuses on semantic errors detectable through execution; it may not help with subtle logical errors that pass tests but violate unstated requirements. (8) Negative examples might bias the model toward overly conservative code generation - we will monitor for this through diversity metrics.
