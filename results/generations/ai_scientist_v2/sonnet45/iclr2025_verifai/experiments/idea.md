## Name

spec_aware_probabilistic_decoding

## Title

Specification-Aware Probabilistic Decoding: Soft Verification Constraints for LLM Code Generation

## Short Hypothesis

By computing continuous specification satisfaction scores during decoding and blending them with LLM token probabilities, we can guide code generation toward correct implementations without the brittleness of hard constraint pruning. This approach handles partial specifications and enables graceful trade-offs between fluency and correctness, which is particularly valuable when complete formal specifications are unavailable or when hard constraints would over-constrain the generation space.

## Related Work

Recent work on constrained decoding for code generation uses hard constraints: type-constrained decoding (Mündler et al., 2025) enforces type correctness by pruning type-invalid continuations using prefix automata, grammar-constrained decoding uses CFGs/DFAs to ensure syntactic validity (ABS, Collura et al., 2025), and vulnerability-constrained decoding (Storhaug et al., 2023) filters vulnerable patterns. These approaches make binary decisions (valid/invalid) and fail when specifications are partial, ambiguous, or too complex for decidable enforcement. Our work differs fundamentally by using soft constraints: we compute continuous specification satisfaction scores and blend them with LLM probabilities, enabling graceful degradation and handling of incomplete specifications. This is not a trivial extension—it addresses a different problem class where hard constraints are infeasible or undesirable.

## Abstract

Large language models have shown remarkable code generation capabilities, but ensuring correctness remains challenging. Existing constrained decoding methods enforce hard constraints through pruning (e.g., type systems, grammars), which fail when specifications are partial or too complex for decidable enforcement. We propose Specification-Aware Probabilistic Decoding (SAPD), a novel approach that uses lightweight formal specifications to compute continuous satisfaction scores for candidate token continuations and blends these with LLM probabilities during decoding. Unlike binary filtering, SAPD enables graceful trade-offs between specification adherence and generation fluency. We implement SAPD using symbolic execution to score partial programs against pre/post-conditions and invariants, then combine scores with LLM probabilities via a tunable blending parameter. Experiments on HumanEval and MBPP benchmarks augmented with specifications show that SAPD improves specification satisfaction rates by 35-50% over vanilla generation while maintaining code quality, and outperforms hard-constraint baselines by 20-30% when specifications are partial or ambiguous. We analyze the trade-off between specification weight and generation quality, demonstrating that soft constraints are particularly valuable for real-world scenarios with incomplete formal specifications. Our approach opens a new direction for integrating formal methods into neural code generation through probabilistic guidance rather than deterministic filtering.

## Experiments

1) Dataset Preparation: Augment HumanEval and MBPP with lightweight formal specifications (pre/post-conditions, simple invariants) using Dafny-style syntax. Create three specification completeness levels: full (complete specs), partial (some conditions missing), and weak (only high-level properties). 2) Implementation: Integrate symbolic execution engine (e.g., Z3-based) to compute satisfaction scores for each beam candidate. Score computation: for each token continuation, perform lightweight symbolic execution on partial code to estimate how many specification clauses are satisfied/violated. Blend scores with LLM probabilities: P_final = α * P_LLM + (1-α) * normalize(score_spec), where α is tunable. 3) Baselines: vanilla LLM (GPT-3.5, CodeLlama), type-constrained decoding (Mündler et al. method), post-hoc verification with regeneration, hard specification pruning (prune any continuation that violates any spec clause). 4) Evaluation Metrics: specification satisfaction rate (% of generated code passing all specs), partial satisfaction rate (% of specs satisfied even if not all), pass@k on test cases, code quality metrics (compilation rate, execution success), generation latency. 5) Ablation Studies: vary α from 0 to 1 to analyze fluency-correctness trade-off, compare performance across specification completeness levels, measure impact of symbolic execution timeout on generation speed. 6) Analysis: qualitative analysis of cases where soft constraints succeed but hard constraints fail, measure correlation between specification scores and actual correctness.

## Risk Factors And Limitations

1) Symbolic execution overhead: computing satisfaction scores may slow generation significantly. Mitigation: use lightweight abstract interpretation or timeout-based approximations. 2) Specification quality dependency: poor specifications may misguide generation. However, this is inherent to any specification-based approach. 3) Score normalization challenges: different specifications may have incomparable score ranges. Will need careful normalization strategies. 4) Limited to decidable properties: very complex specifications may be too expensive to score during generation. Will focus on tractable property classes (simple contracts, type properties, basic invariants). 5) Hyperparameter sensitivity: the blending parameter α may require task-specific tuning. Will provide guidelines based on specification completeness. 6) Evaluation complexity: need to create specification-augmented benchmarks, which is labor-intensive. Will start with existing benchmarks and add specifications semi-automatically using LLMs with manual verification.
